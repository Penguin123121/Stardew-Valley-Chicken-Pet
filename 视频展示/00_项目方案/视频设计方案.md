# 《星露谷小鸡桌宠》Bilibili 介绍视频设计方案

## 1. 项目目标

制作一条横屏 16:9 的 Bilibili 项目介绍视频，通过轻松幽默、像素风包装的方式介绍“星露谷小鸡桌宠”的灵感来源、已实现功能、GitHub 下载方式、制作过程及版权边界。

视频无语音解说，依靠动态字幕、背景音乐、低频后期音效、鼠标高亮、局部放大和原创字幕梗推进。粗剪完成后先由主包审核，再进入精剪和最终导出。

## 2. 已确认规格

- 成片比例：16:9。
- 最终分辨率：1920×1080。
- 最终帧率：60 FPS。
- 原始录制：2560×1600，保持 Windows 推荐的 150% 显示缩放。
- 后期画面处理：从 2560×1600 居中裁切为 2560×1440，再缩放导出为 1920×1080。
- 时长：以内容完整为准，最长不超过 5 分钟；目标约 3 分 30 秒至 4 分 30 秒。
- 剪辑软件：剪映专业版 10.9.0。
- 录屏工具：Windows 11 截图工具；若无法录制游戏系统声音，游戏开头改用 Windows 自带 Xbox Game Bar 单独录制。
- 交付方式：先交付剪映粗剪工程和 MP4 预览版，主包确认后再精剪。
- 视频称呼：使用“主包”称呼项目发起者。
- 配音：无配音。

## 3. 创意方向

### 3.1 核心故事线

采用“小鸡从星露谷跑进主包桌面”的故事结构：

1. 《星露谷物语》标题界面建立世界观。
2. 小鸡像素化“越狱”到真实 Windows 桌面。
3. 展示桌宠的日常行为和互动功能。
4. 引导观众通过 GitHub Release “领养同款”。
5. 说明主包与 AI 工具的协作过程。
6. 以小鸡逃跑收尾，邀请观众留言建议。

### 3.2 视觉与幽默风格

- 风格：轻松吐槽型。
- 包装：像素块、窗口缩放、速度线、emoji、箭头、高亮框和局部放大。
- 笑点密度：约每 10–15 秒一个，不遮挡功能信息。
- 梗图策略：以原创字幕梗、emoji 和画面放大为主；仅少量使用来源明确的外部素材。
- 字幕：每屏原则上不超过两行，关键操作字幕至少停留 1.5 秒。

## 7. 音效误解防护

桌宠当前没有内置声音功能。视频第一部分开始时在角落显示约 3 秒：

> 本视频音效为后期剪辑添加，桌宠当前暂无内置音效功能。  
> 音响系统：还在鸡蛋里 🥚

声音规则：

- 背景音乐持续低音量播放。
- 不添加脚步、咀嚼、抓取、点击、滑块刻度等容易被理解为程序自带的拟真操作音效。
- 仅在章节标题、梗字幕、局部放大、箭头和少数夸张转场处使用明显属于剪辑语言的音效。
- 平均每 10–15 秒最多一个音效，保留纯 BGM 段落。
- 结语可再次简短标注“音效为视频后期添加”。

## 8. 第二部分：GitHub 下载和运行

演示路径：

1. 打开 `github.com/Penguin123121/Stardew-Valley-Chicken-Pet`。
2. 浏览 README 中的功能、使用方法和常见问题。
3. 放大右侧 `Releases`。
4. 进入最新 Release。
5. 定位 `Assets`，提醒不要误点 `Source code`。
6. 点击 `.exe` 并展示下载进度。
7. 打开下载文件夹并定位 `.exe`。
8. 双击运行。
9. 如出现 SmartScreen，完整展示“更多信息 → 仍要运行”。
10. 展示桌宠成功启动。

安全字幕：

> 请仅从本项目官方 GitHub Release 下载。首次运行可能出现“未知发布者”提示；继续运行前请核对仓库地址与文件来源，不要对来源不明的 `.exe` 使用“仍要运行”。

录制和剪辑时必须隐藏浏览器账号、书签、扩展、Windows 用户名、下载路径和其他私人文件。不能把 SmartScreen 描述为无意义提示，也不能伪造安全认证。

## 10. 音乐设计

- 开头：保留《星露谷物语》标题界面录制时的现场原声。
- 第一部分：活泼俏皮、节奏较强的免版权像素风音乐。
- 第二部分：轻快但规整、存在感较低的免版权像素风音乐。
- 第三部分：温暖、有收束感的免版权像素风音乐。
- 所有免版权素材必须记录来源、作者、许可类型、下载地址和下载日期。
- 不把“免版权”理解为“无需署名”；严格按具体许可执行。

## 11. 原始素材规格和安全区域

- 所有素材保持笔记本原生 2560×1600、150% 缩放录制。
- 关键内容放在居中的 2560×1440 安全区域内。
- 上下各约 80 像素属于默认裁切区，关键按钮、小鸡和鼠标操作不得长期停留于此。
- 托盘菜单镜头允许单独调整画面裁切位置，以完整展示任务栏区域。
- 游戏标题界面录制后单独检查 Logo 和菜单是否被 16:9 裁切影响。
- 每个动作前后保留 3–5 秒余量。

## 13. 素材检查与异常处理

### 13.1 录制前测试

先录制 `00_录制测试.mp4`，检查：

- 文件是否为 2560×1600；
- 画面裁成 16:9 后关键内容是否完整；
- 缩小至 1080p 后文字是否清晰；
- 鼠标移动和小鸡动画是否流畅；
- 游戏系统声音能否被录入；
- 是否存在通知声、个人信息或私人文件。

若截图工具无法录制游戏系统声音，仅将游戏标题界面改用 Xbox Game Bar 录制，其他无声操作仍用截图工具。

### 13.2 粗剪前检查

- 核心互动均至少有一条完整可用素材；
- 小鸡没有被窗口遮挡；
- GitHub 地址、Release、Assets 和 `.exe` 文件名清晰；
- SmartScreen 过程完整；
- 没有账号、聊天、真实姓名、用户名和私人路径泄露；
- 素材损坏、卡顿或缺镜头时，先输出精确补录清单，不用低质量镜头强行剪辑。

## 14. 工程交付和审核

粗剪阶段交付：

- 剪映专业版 10.9.0 草稿；
- 1920×1080、60 FPS 的 MP4 预览；
- 素材缺口说明；
- 带时间点的审核清单。

主包确认内容顺序、节奏、字幕和音乐方向后，再制作动态字幕、像素转场、原创梗图、音效混音和最终版权卡。

## 15. 发布前建议项

仓库 README 当前声明使用 MIT License，但现有文件清单中未发现独立 `LICENSE` 文件。正式发布视频前建议：

1. 添加标准 MIT `LICENSE` 文件，仅覆盖项目原创代码。
2. 在 README 中明确第三方游戏名称、精灵图、音乐和其他素材不随 MIT 许可转授。
3. 在视频简介中列出游戏权利归属、项目仓库、音乐音效来源及许可。

这些属于后续独立的仓库修改建议，未经主包确认不直接实施。

## 16. 成功标准

- 观众能在不听配音的情况下理解项目用途和核心功能。
- 观众能准确找到官方 GitHub Release 并理解安全运行步骤。
- 观众不会误以为桌宠当前自带声音功能。
- 观众不会误以为项目由 ConcernedApe 官方制作或背书。
- 主包、Claude Code 和 Codex 的实际分工表述准确。
- 视频节奏轻松幽默，但下载说明和版权信息清晰、严谨。
- 粗剪工程能在剪映专业版 10.9.0 中正常打开，预览 MP4 音画同步。
